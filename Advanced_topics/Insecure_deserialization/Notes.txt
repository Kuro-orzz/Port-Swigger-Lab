Serialization object --> Stream of bytes --> file, db, memory --> Stream of bytes -> Deserialization object

Serialization: 
- Process of converting complex data structures into flat format that can be sent and received as a sequential stream of bytes
- Help:
    + Write complex data to inter-process memory, file or database
    + Send complex data (Ex: over a network, between different components, API call,..)
Deserialization:
- Process of restoring byte stream to a fully functional replica of original object

How object are serialized depend on the language:
- Some serialize into binary formats
- Whereas others use different string formats
- Varying degrees of human readability

Note:
- All original object attribute are stored in serialized data stream, include any private fields
- To prevent field from being serialized, it must be explicity marked as "transient" in class declaration
- This maybe referred to as marshalling (Ruby) or pickling (Python)

Insecure deserialization:
- Known as an "object injection" vuln
- User controllable data is deserialized by website, enable attacker manipulate serialized objects
- Possible replace serialized object with an object of different class
    + Might cause an exception
- Many deserialization-base attacks are completed before deserialization is finished

How insecure deserialization vuln arise:
- Lack of understanding how dangerous deserializing user input. Ideally, should never deserialized at all
- Web owner think they safe because they implement some form of additional check on the deserialized data.
    + Often ineffective, virtually impossible to implement validation or sanitization to account for every eventuality
- Deserialized object are often assumed to be trustworthy
    + Even if binary serialization format, it's possible for attacker to exploit, just require more effort
- Hard to predict which method can be invoked on the malicious data
    + Chain together a long series of unexpected method invocation
    + Passing data into a sink that completely unrelated to initial source
    -> Impossible to anticipate the flow of malicious data and plug every potential hole

Impact:
- Provide entry point to a massively increased attack surface
- Allow attacker to reuse existing code in harmful way, often RCE
- Even RCE not possible, it still can lead to privilege escalation, arbitrary file access, DOS attack

Exploiting insecure deserialization vulnerabilities:
- Example from PHP, Ruby, Java deserialization
- How to identify:
    + It is simple whether whitebox or blackbox testing
    + During auditing, look at all data being passed into website and try identify anything that look like serialized data
    + Easily identified if you know format that different language use.
    + Tip: Burp Scanner automatically flag any HTTP messages that appear to contain serialized object
    + PHP serialization format:
        $user->name = "carlos";
        $user->isLoggedIn = true;
        Serialized: O:4:"User":2:{s:4:"name":s:6:"carlos";s:10:"isLoggedIn":b:1;}
        + Native method are serialize() and unserialize()
    + Java serialization format:
        + Use binary serialization formats, more difficult to read
        + Can still identify if you know how to recognize a few tell-tale signs
            + Always begin with the same bytes, 'ac ed' in hexadecimal and rO0 in Base64
        + Class that implement the interface 'java.io.Serializable' can be serialized and deserialized
        + If have source code, try to find readObject(), used to read and deserialize data from InputStream
- Manipulating serialized objects:
    + Modifying object attributes: (1)
        + Consider website that uses a serialized User object to store data about user's session in cookie. If attacker spotted this, they might decode to get serialized object.
        + Then modify value to privilege escalation
        Example: 
            $user = unserialize($_COOKIE);
            if ($user->isAdmin === true) {
            // allow access to admin interface
            }
    + Modifying data types: (2)
        + PHP logic is particularly vuln to this because of operator (==) when comparing different data types.
            Example: 
                5 == "5" => true (string->integer)
                5 == "5 of something" --> 5 == 5 => true (string->integer)
                (base on the initial number, the rest will be ignored)
                0 == "Example string" => true (string->integer)
                (PHP 7.x and earlier)
        + If attacker modified password to integer 0 instead of string and password does not start with a number, it will return true
            Example:
                $login = unserialize($_COOKIE)
                if ($login['password'] == $password) {
                // log in successfully
                }
            Note: only possoble because deserialization preserves the data typek, if fetch password from request directly, 0 would be convert to string
- Using application functionality: (3)
    + Might perform dangerous operation on data from a deserialized object
    + Deleting their own user account would then delete this arbitrary file as well.
    Example:
        User delete -> user's picture profile delete by filepath attribute
    + Could use this to delete other file by modify the path
- Magic methods:
    + Special subset of methods, that are invoked automatically whenever a particular event or scenario occurs
    + Common feature of OOP, can be custom by dev to execute any code they want
    + Example:
        PHP: __construct()
        Python: __init__
    + Can become dangerous when the code they execute handles attack-controllable data
        Example: from a deserialized object
    + Some language have magic method invoked automatically during deserialization process
        Example: 
            PHP: unserialize() look for and invokes onject __wakeup() magic method
            Java: ObjectInputStream.readObject() read data from initial byte steam and acts like a constructor for "re-initializing" a serialized object --> calling magic method re-initial
- Injecting arbitrary objects: (4)
    + In OOP, method available are determined by its class
    + If attacker can manipulate class of object being passed in as serialized data, they can influence what code is executed after and even during deserialization
    + The fact that, object is not of the expected class does not matter, it might cause exception, but malicious object will already be instantiated by then
    + Can used to be initiate more complex attack involving long series, known as "gadget chanin"
- Gadget chains:
    + "Gadget" is a snippet of code exists can help attacker achieve a particular goal
    + Attacker's goal: might invoke a method that pass their input into another gadget
    + By chaining multiple gadgets together, attacker can pass input into a dangerous "sink gadget", where can cause maximum damage
    + Unlike other types of exploit, all codes exist on website, only thing attacker controls is the data passed into gadget chain, done using magic method, known as "kick-off gadget"
    + Working with pre-built gadget chains: (5)
        + https://github.com/frohoff/ysoserial
        + Manually identify almost impossible without source code access.
        + Fortunately, there are several tools provide pre-discovered chains that have been successfully exploited (even if you don't have access to source code)
        + ysoserial: tool for Java deserialization, provided gadget chains for library that application is using. It create serialized object based on the selected chain. This still involves a certain amount of trial and error, but it is considerably less labor-intensive than constructing your own gafget chains manuallly
            Example: Java 16 and above
                java \
                    --add-opens=java.xml/com.sun.org.apache.xalan.internal.xsltc.trax=ALL-UNNAMED \
                    --add-opens=java.xml/com.sun.org.apache.xalan.internal.xsltc.runtime=ALL-UNNAMED \
                    --add-opens=java.base/java.net=ALL-UNNAMED \
                    --add-opens=java.base/java.util=ALL-UNNAMED \
                    -jar ysoserial-all.jar [payload] '[command]'
        + Detect insecure deserialization on virtually any server:
            + URLDNS: trigger DNS lookup. It does not rely on specific vuln library and works in any known Java version
                Example: java -jar ysoserial.jar URLDNS 'http://YOUR.burpcollaborator.net'
            + JRMPClient: cause server to try establishing a TCP connection to the supplied IP address (need provide raw IP rether than hostname), useful in enviroment where all outbound traffic is firewalled. You can try gen payload with 2 different IP (local and external). If response immediately for a local, but hangs for external, causing delay in response. The subtle time different in reponse can help you detect whether deserialization occurs on server, even in blind cases
                Example:
                    java -jar ysoserial.jar JRMPClient '192.168.1.1'  # local IP
                    java -jar ysoserial.jar JRMPClient '1.2.3.4'      # external
        + PHP Generic Gadget Chains: (6)
            + https://github.com/ambionics/phpggc
    + Working with documented gadget chains: (7)
        + Not always be a dedicated tool available for exploiting
        + You will need to looking online to see document exploit can adapt manually

- Creating your own exploit: (8, 9)
    + Need when off-the-shelf gadget chains and documented exploit are unsuccessful
    + To build it, you cartainly need source code access
        + First step is to identify class contain magic method that invoke during deserialization, see if it does anything dangerous to user-controllable attributes
        + If is not exploitable on its own, can serve as "kick-off gadget", study any method that invoked by this, look at each methods that subsequently invoke
        + Repeat process until dead end or identify a dangerous sink gadget into which you controllable data is passed
        + Construct your own serialized object contain payload
        + Look out for opportunities to use this extra attacker surface to trigger secondary vuln
