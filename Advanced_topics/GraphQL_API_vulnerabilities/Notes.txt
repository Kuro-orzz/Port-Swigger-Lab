GraphQL vulnerabilities:
- Arise due to implementation and design flaw
- Example: introspection feature may be left active, enabling attackers to query API to glean information about its schema

Finding GraphQL endpoints:
- Universal queries:
    + If send query{__typename}, response will include {"data": {"__typename": "query"}}
    + It work because every graphql has a reserved field called __typename return object as string
- Common endpoint names:
    + /graphql
    + /api
    + /api/graphql
    + /graphql/api
    + /graphql/graphql
    + if these not work, append /v1 to path
    + Note: graphql services often response to any non-GraphQL with a "query not present" or similar error
- Request methods:
    + Should only accept POST method have content-type application/json
    + Some endpoint may accept alternative GET method or POST that use content-type of x-www-form-urlencoded

Exploiting unsanitized arguments
Discovering schema information:
- Best way to do is using introspection queries, query for information about schema
- Using introspection:
    + { "query": "{__schema{queryType{name}}}" }: return names of all available queries
    + Full introspection: returns full details on all queries, mutations, subscriptions, types, and fragments.

    query IntrospectionQuery {
        __schema {
            queryType {
                name
            }
            mutationType {
                name
            }
            subscriptionType {
                name
            }
            types {
             ...FullType
            }
            directives {
                name
                description
                args {
                    ...InputValue
            }
            onOperation  #Often needs to be deleted to run query
            onFragment   #Often needs to be deleted to run query
            onField      #Often needs to be deleted to run query
            }
        }
    }

    fragment FullType on __Type {
        kind
        name
        description
        fields(includeDeprecated: true) {
            name
            description
            args {
                ...InputValue
            }
            type {
                ...TypeRef
            }
            isDeprecated
            deprecationReason
        }
        inputFields {
            ...InputValue
        }
        interfaces {
            ...TypeRef
        }
        enumValues(includeDeprecated: true) {
            name
            description
            isDeprecated
            deprecationReason
        }
        possibleTypes {
            ...TypeRef
        }
    }

    fragment InputValue on __InputValue {
        name
        description
        type {
            ...TypeRef
        }
        defaultValue
    }

    fragment TypeRef on __Type {
        kind
        name
        ofType {
            kind
            name
            ofType {
                kind
                name
                ofType {
                    kind
                    name
                }
            }
        }
    }

    + Note: if query doesn't run, remove onOperation, onFragment, onField from the query

Bypassing GraphQL introspection defenses:
- Technique 1: insert a special character after __schema keyword
    + When dev disable introspection, they use regex to exclude __schema in queries
        Example: if "__schema{" in query -> block
    + Insert endline { "query": "query{__schema\n{queryType{name}}}" } -> bypass
    + Some special char: spaces, new lines, commas,...
- Technique 2: change to GET method
    + As introspection may only disable over POST method
    + Try a GET request, or a POST request with a content-type of x-www-form-urlencoded
    + Example: GET /graphql?query=query%7B__schema%0A%7BqueryType%7Bname%7D%7D%7D

Bypassing rate limiting using aliases:
- Ordinarily, GraphQL objects can't contain multiple properties with same name
- Aliases enable to bypass restriction by explicity naming properties you want API return
- Example:
    + Invalid: Same name
        query {
            getUser(id: 1) { username }
            getUser(id: 2) { username }
        }
    + Valid: Aliases
        query {
            user1: getUser(id: 1) { username }
            user2: getUser(id: 2) { username }
        }
- Some rate limit work based on the number of HTTP requests received rather than number of operations performed on the endpoint
    + Without alias, 100 try need 100 HTTP request
    + With alias, 100 try may need only 1 HTTP request

GraphQL CSRF:
- This can be used as a vector for CSRF attack
- When vulnerabilities arise: accept GET request or POST with content-type x-www-form-urlencoded or not contain CSRF token
- application/json is safe, because this is a non-simple request so blocked by CORS
