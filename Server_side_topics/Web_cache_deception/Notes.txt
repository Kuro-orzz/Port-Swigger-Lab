Web cache deception:
- Trick a web cache into storing sensitive, dynamic content.

Different between web cache deception -- web cache poisoning:
- Web cache deception:
    Attacker send malicious url -> User click -> Server return sensitive data -> Cache sensitive data -> Attacker go to cache to get sensitive info
- Web cache poisoning:
    Attacker send malicious content to server -> Server response contain it -> Cache and send to users

Cache keys:
- Includes the URL path and query parameters, can also include a variety of other elements like headers and content type.
- If the incoming request's cache key matches that of a previous request, the cache considers them to be equivalent and serves a copy of the cached response.

Cache rules:
- Cache for how long, static resource
- Static file extension rules: for example '.css' for stylesheets or '.js' for JavaScript files.
- Static directory rules: for example '/static' or '/assets'.
- File name rules: such as 'robots.txt' and 'favicon.ico'.

Constructing a web cache deception attack:
- Step 1: Identify endpoint return sensitive information, review response
- Step 2: Identify a discrepancy in how the cache and origin server parse the URL path. How they
    + Map URLs to resources.
    + Process delimiter characters.
    + Normalize paths.
- Step 3: Generate malicious url to trick the cache into storing a dynamic response -> Wait user access the url -> Go to that url to get sensitive information

- Using a cache buster:
+ Make sure that each request you send has a different cache key
+ Use param miner, select Add dynamic cachebuster, view in Logger
- Detecting cached responses:
+ Look at response headers and response times
+ 'X-Cache' header provide info about whether it's serve from the cache
    Hit: server from cache
    Miss: cache not contain request key, fetched from server
    Dynamic: origin server dynamic generated the content
    Refresh: cache content outdated and needed to be refreshed or revalidated
+ 'Cache-Control' header may include a directory that caching, like 'public' with 'max-age' > 0

How to prevent:
- Always use Cache-Control headers to mark dynamic resources, set with the directives no-store and private.
- Configure your CDN settings so that your caching rules don't override the Cache-Control header.
- Activate any protection that your CDN has against web cache deception attacks. Many CDNs enable you to set a cache rule that verifies that the response Content-Type matches the request's URL file extension. For example, Cloudflare's Cache Deception Armor.
- Verify that there aren't any discrepancies between how the origin server and the cache interpret URL paths.
