Server-side parameter pollution:
- It occurs when web embeds user input in a server-side request to an internal API without adequate encoding.
+ Override existing parameters.
+ Modify the application behavior.
+ Access unauthorized data.

Testing for server-side parameter pollution in the query string:
- Truncating query strings: using &, #, =, ?
    Client request: ?name=peter%23foo&back=/home
    Server request: ?name=peter#foo&publicProfile=true
    + If server return 'peter' user, request may have been truncated
- Injecting invalid parameters:
    ?name=peter%26foo=xyz&back=/home
    ?name=peter&foo=xyz&publicProfile=true
    + If same response, it was successfully injected but ignored by application
- Injecting valid parameters:
    + If you know a hidden param, add it to query and check response
    ?name=peter%26email=foo&back=/home
    ?name=peter&email=foo&publicProfile=true
- Overriding existing parameters:
    ?name=peter%26name=carlos&back=/home
    ?name=peter&name=carlos&publicProfile=true
    + Different behavior between language:
        PHP: parses last parameter only, search for "carlos"
        ASP: combine both parameters, search for "peter,carlos"
        Node.js: parses first parameter, search for "peter"

- Description:
+ First just send reset token to administrator
+ Then check forgotPassword.js, static file, you will see it contains param named reset_token
+ Try:
    + Adding %26x=y and check response, if "Parameter is not supported", it means BE decode %26->&
    + Adding %23 and check response, if "Field not specified", it means BE have been truncated by %23-># and it also have param named Field
    + Try to find hidden param using Intruder, username and email is email is return 200
    + Check different response between that, if field=username, it return username and if field=email, it return email
    + So we can get reset_token by adding field=reset_token
+ Use that to reset admin password, then login and delete target account

How to prevent:
- Secure your documentation if you don't intend your API to be publicly accessible.
- Ensure your documentation is kept up to date so that legitimate testers have full visibility of the API's attack surface.
- Apply an allowlist of permitted HTTP methods.
- Validate that the content type is expected for each request or response.
- Use generic error messages to avoid giving away information that may be useful for an attacker.
- Use protective measures on all versions of your API, not just the current production version.
