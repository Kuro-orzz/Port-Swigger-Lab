Some types of logic flaw base on the initial mistakes that introduced the vulnerability in the first place:
- Excessive trust in client-side controls (1)
- Failing to handle unconventional input (2, 5, 6)
- Making flawed assumptions about user behavior
+ Trusted users won't always remain trustworthy (change email after register) (3)
+ Users won't always supply mandatory input
    + Try to remove param (7)
    + Users won't always follow the intended sequence (8, 9)
- Domain-specific flaws (4, 10)
- Providing an encryption oracle
- Email address parser discrepancies


01_Excessive_trust_in_client_side_controls
- Use burp proxy to catch request -> modify param -> vuln
- Description:
+ Just a simple lab, flaw logic, only authentication in client
+ When checkout, look at request, we can modify price param to make it lower than balance

02_High_level_logic_vulnerability
- Use burp proxy to catch request -> modify param -> vuln
- Description:
+ This lab has logic flaw, allow user to fill quantity negative number
+ Just buy target product with quantity=1, then buy other product with negative quantity to make total money lower than balance
+ Note: cuz this only allow total price positive number before checkout 

03_Inconsistent_security_controls
- Register a new account -> Login and change email to admin address -> And you got admin panel
- Description:
+ This lab has flaw logic in changing email after register
+ We can just register, login, then change email to admin address to get admin panel

04_Flawed_enforcement_of_business_rules

- Description:

05_Low_level_logic_flaw

- Description:

06_Inconsistent_handling_of_exceptional_input
- Try to truncate the email -> If truncate success and tail is @target.com -> Login and go to admin panel -> Delete target account
- Description:
+ First we register using long string mail follow by tail @dontwannacry.com.exploit-EXPLOIT_ID.exploit-server.net
+ Cuz limit is 255, trying to register with email longer than that, make it truncate and save with tail @dontwannacry.com
+ Now login as usual and we got access to admin panel.

07_Weak_isolation_on_dual_use_endpoint
- Try to change password -> Remove current_password field, change username to admin -> Login admin and delete target account
- Description:
+ After login, we can use the change password api to solve lab
+ Just remove current_password field, it still work normally
+ Try replace username by admin, it still work without authentication
+ Change admin password and login

08_Insufficient_workflow_validation
- Add product to cart -> Try to checkout but not allow to redirect -> Skip fail page and go to success bought product
- Description:
+ Login and add product to cart
+ When checkout cart, not allow redirect to fail page (skip step)
+ Try to use other request and redirect to success page

09_Authentication_bypass_via_flawed_state_machine

- Description:
+ Login normal account, it will auto redirect you to /role-selector
+ Cuz default role admin, you just have to skip step and access admin panel
