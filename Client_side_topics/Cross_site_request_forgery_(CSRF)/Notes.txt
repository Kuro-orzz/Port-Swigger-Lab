CSRF:
- It allows an attacker to partly circumvent the same origin policy, which is designed to prevent different websites from interfering with each other.

3 key conditions csrf attack possible:
- A relevant action
- Cookie-based session handling
- No unpredictable request parameters

Generate CSRF PoC online: https://csrfshark.github.io/app/

If only need GET method, payload below:
    <img src="https://vulnerable-website.com/email/change?email=pwned@evil-user.net">

XSS vs CSRF:
- XSS: allow attacker to execute arbitrary Javascript in victim's browser
+ Two way vulnerability, attacker received response, exfiltrate data and reuse it
- CSRF: allow attacler to induce victim user perform action that they do not intend to
+ One way vulnerability, attacker can reduce victim to issue an HTTP request but can't retrieve the response

Common defences against CSRF:
- CSRF tokens: unique, secret, unpredictable value generated by server-side and shared with client
- SameSite cookies: browser security mechanism determines when website's cookies are included in request originating from other websites
    + Chrome enforces Lax SameSite restritions by default
- Referer-based validation: verify request originated from application's own domain (weak validation)

Note: csrf token don't have to be sent as hidden param in POST request, some applications place csrf in HTTP headers

Validation of CSRF token depends on request method:
- Can bypass by replace POST by GET and put parameter in url
    GET /email/change?email=pwned@evil-user.net HTTP/1.1
Validation of CSRF token depends on token being present:
- Some applications skip validation if the token is omitted (removed param)
CSRF token is not tied to the user session:
- Some applications do not validate that token belong to the same session as the user who making request
- Maintains a global pool of tokens that it has issued and accepts any token that appears in this pool.
CSRF token is tied to a non-session cookie:
- Some applications do tie the CSRF token to a cookie, but not to the same cookie that is used to track sessions.
- Can reuse other user csrf token with csrfKey belong to that, application validate it's true, so pass
- Note: 
    + Any applications within the same dns, can be setting and share other subdomain db and can use reuse same cookies
CSRF token is simply duplicated in a cookie:
- Some applications do not maintain any server-side record of tokens that have been issued
- Duplicate each token within a cookie and a request parameter
    + Application only verifies that the token submitted in the request parameter matches the value submitted in the cookie

Bypassing SameSite cookie restrictions:
- Normal TLD: .com, .net, .org (only contain 1 part)
- eTLD (effective TLD): .co.uk, .com.vn, .gov.au, .ac.jp (multiple parts)
- Same-site: same scheme, eTLD+1
- Same origin: same scheme, domain, port
- Same-site restriction levels: Strict, Lax, None
+ Can config level for each cookie by include 'SameSite' attribute in the 'Set-Cookie' 
+ SameSite=Strict: will not include cookies in any cross-site request, most secure option, but can negatively impact user experience in cases where cross-site functionality is desirable
+ SameSite=Lax: only include cookies if request is GET method and request from a top-level navigation by user(click on link), if it initiated by scripts, iframes, or references images and other other resource, cookie won't be include
+ SameSite=None: disable SameSite restriction (other browser except Chrome default is None)
    When using None, must also include 'Secure' attribute, to ensure cookie only sent in encrypted message over HTTPS (avoid HTTP and MITM)

Bypassing SameSite Lax restrictions using GET requests:
- If browser use Lax, we still be able to perform CSRF attack by GET request
Bypassing SameSite restrictions using on-site gadgets:
- This case, SameSite=Strict, won't include cookies in any cross-site request
- You must find client-side url that has redirect in param
- This is not possible with server-side redirects, it will recognize that request follow redirect resulted from a cross-site request, still apply cookies restrictions
Bypassing SameSite restrictions via vulnerable sibling domains:
- Vuln can come from sub domain, it's still be same-site even if it's issued cross-origin
- XSS can compromise site-based defenses completely, exposing all of the site's domains to cross-site attack
Bypassing SameSite Lax restrictions with newly issued cookies:
- If site apply Lax by default, to avoid braking single sign-on (SSO) mechanisms
    + It doesn't actually enforce these restriction for the first 120s on POST request
    + So there is 2 minutes window user may be susceptible to cross-site attacks
- Process:
    + Redirect victim to endpoint that could reset cookies and disable SameSite policy during that time
    + Wait for victim click (bypass pop up)
        Example: this script blocked by default window.open('https://vulnerable-website.com/login/sso');
                but if add onclick event, will allow pop up window.onclick = () => { .... }
    + CSRF attack during SameSite disable time

Bypassing Referer-based CSRF defenses:
- Some application use Referer header + csrf token to defend against csrf attacks
- Validation of Referer depends on header being present:
    + Validate when it is present in request but skip if header is omitted
        <meta name="referrer" content="never">
        + never, no-referer: not send referer header
        + origin: only send domain, not full path
        + unsafe-url: full url (not strip)
- Validation of Referer can be circumvented:
    + Some applications validate in naive way that can be bypassed, check start with expected value
        http://vulnerable-website.com.attacker-website.com/csrf-attack
    + If validate contain it own domain name
        http://attacker-website.com/csrf-attack?vulnerable-website.com
    + Although may be able to identify using Burp, but no longer work when test PoC in browser, many browser strip query string from Referer by default
    + But can add 'Referrer-Policy: unsafe-url' and then referer not gonna strip anyway
