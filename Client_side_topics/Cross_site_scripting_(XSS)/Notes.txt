3 types XSS:
- Reflected XSS: from current HTTP request
- Stored XSS: from website database
- DOM-based XSS: vuln exist in client-site code rather than server-side code

Used for:
- Impersonate or masquerade as the victim user.
- Carry out any action that the user is able to perform.
- Read any data that the user is able to access.
- Capture the user's login credentials.
- Perform virtual defacement of the web site.
- Inject trojan functionality into the web site.

Payload:
- <script>alert(1)</script>
- <svg onload=alert(1)>
- <img src=1 onerror=alert(1)>

Reflected XSS:
- <script>/* Bad stuff here... */</script>
- Can be found quickly and reliably using Burp Suite scanner.
- Finding process:
+ Test every entry point: parameters, data within the url query string, message body, HTTP headers
+ Determine the reflection context: Each location, check if input value appear
+ Test different payload

Stored XSS:
- Same as reflected XSS, but it store malicious data in db and use it later to trigger
- Any out-of-band can allow attacker to deliver data into application
- Test locate link between entry and exit points

DOM-based XSS:
- Arise when JS takes data from attacker and pass it to a sink that support dynamic code execution (eval(), innerHTML). Typically accessed with 'window.location' object
- Note: 
+ Chrome/Firefox/Safari auto URL-encode location.search, location.hash, but old IE11/Edge is not -> if data is encoded before process, attacker is harder
+ In Chrome's developer tools, you can use Control+Shift+F to search all the page's JavaScript code for the source.
- Exploiting DOM XSS with different sources and sinks:
+ Payload:
    document.write('... <script>alert(document.domain)</script> ...');
    element.innerHTML='... <img src=1 onerror=alert(document.domain)> ...'
+ Sources and sinks in third-party dependencies
    + DOM XSS in jQuery: (5, 6)
        + 'attr()' function can change attribute of DOM
            $(function() {
                $('#backLink').attr("href",
                    (new URLSearchParams(window.location.search)).get('returnUrl')
                );
            });
            <a id="backLink" href="javascript:alert(1)">Back</a>
        + '$()' use selector function with 'location.hash' for animation and auto-scrolling to particular element on page, often implemented using a vulnerable 'hashchange' event
            $(window).on('hashchange', function() {
                var element = $(location.hash);
                element[0].scrollIntoView();
            });
            <iframe src="https://vulnerable-website.com#" onload="this.src+='<img src=1 onerror=print()>'">
    + DOM XSS in AngularJS: (11)
        + https://breachforce.net/portswigger-xss-dom-angularjs
        + When a site uses the ng-app attribute on an HTML element
        + Possible to execute JavaScript without angle brackets or events
            {{$on.constructor('alert(1)')()}}
- DOM XSS combined with reflected and stored datas: (12, 13)
+ document.cookie and document["cookie"] (https://github.com/antoinenguyen-09/Portswigger-Web-Security-Labs/blob/main/BSCP_Practice_Exam.md)
- Which sinks can lead to DOM-XSS vulnerabilities? https://portswigger.net/web-security/cross-site-scripting/dom-based
